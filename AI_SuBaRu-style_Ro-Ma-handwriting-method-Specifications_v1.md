# すばる式ローマ字 筆記法規格（草案）

本書は、訓令式ローマ字を基底入力として、仮名塊を視覚的に立ち上げる「すばる式ローマ字」の筆記法を定義するための草案である。[^30_1]
目的は、一般的な欧文単語と日本語ローマ字列とを初見の字面で見分けやすくし、仮名単位・仮名塊単位を強く意識させることにある。[^30_1]

## 1. 目的と適用範囲

すばる式ローマ字は、学校教育用の厳密な正書法そのものを置き換える規格ではなく、訓令式ローマ字の入力列をもとに、視覚的な違和感を利用して日本語の音節性を強調する筆記法として設計される。[^30_1]
そのため本規格は、音声転写の忠実さだけでなく、「どこをひと塊の仮名として見せるか」という表記設計を中核に置く。[^30_1]

適用対象は、半角小文字の訓令式ローマ字入力を基本とする日本語表記である。[^30_1]
公開表示では、基本変換による表示と、後段の表示例外とを分離して扱う構成を採る。[^30_1]

## 2. 基本原則

### 2.1 子音先頭の立ち上げ

子音を表す文字で始まる仮名単位は、先頭字のみを大文字化する。[^30_1]
したがって、通常の `CV` や `CyV` は `Ka`、`Si`、`Gyo` のように表記する。[^30_1]

### 2.2 独立母音の立ち上げ

子音を伴わない独立したア行母音は、大文字で表記する。[^30_1]
たとえば、再解釈上で独立母音と判定された `o` や `e` は `O`、`E` のように示す。[^30_1]

### 2.3 撥音 `ん`

撥音は小文字 `n` を基本とし、独立した大文字化対象とはしない。[^30_1]
直後が母音字または `y` の場合のみ `n'` とし、それ以外では `n` を用いる。[^30_1]

### 2.4 促音 `っ`

促音は入力解析上では独立トークンとして保持するが、公開表示では単独文字としては立てず、後続子音を肥大化させる要素として扱う。[^30_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入りの塊として再構成する。[^30_1]

## 3. 入力前提

入力は半角小文字の訓令式ローマ字とする。[^30_1]
空白は語境界として保持し、変換処理はまず語または表記単位の分割から始める。[^30_1]

句読点、中点、括弧、ハイフンを含む文字列は、必要に応じて単なる語ではなく「表記単位」として扱う。[^30_1]
これは、複合語や演出的なまとまりを、空白区切りより上位の単位として保持するためである。[^30_1]

## 4. 処理単位

本規格では、入力文字列を次の順で処理する。[^30_1]

1. 表記単位へ分割する。[^30_1]
2. 表記単位をモーラ風トークン列へ分解する。[^30_1]
3. トークン列を仮名塊へ再束ねする。[^30_1]
4. 基本表示を生成する。[^30_1]
5. 必要に応じて表示例外を適用する。[^30_1]

モーラ風トークンは、少なくとも `CV`、`CyV`、`N`、`Q`、`V` を基本分類とする。[^30_1]
ここで重要なのは、最終表示がトークン列の単純な逐次大文字化ではなく、再束ね後の仮名塊に基づいて決まる点である。[^30_1]

## 5. 仮名塊再解釈

### 5.1 `CVn` / `Vn`

`CV + N + 子音` および `CV + N + 語末` は、`CVn` の一塊として再束ねする。[^30_1]
同様に、`V + N + 子音` および `V + N + 語末` は `Vn` の一塊として再束ねする。[^30_1]

この規則により、`ken` は `Ken`、`han` は `Han`、`en` は `En` のように表記される。[^30_1]
語末 `n` は独立クラスタを作らず、直前塊へ小文字従属させる。[^30_1]

### 5.2 `Q + CV` / `Q + CyV`

促音は単独表示せず、後続塊に吸収する。[^30_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入り塊へ、`Q + CyV` は `kkyo`、`ttya` のような塊へ再束ねする。[^30_1]

### 5.3 長母音と二重母音

`CV + 同母音V` は、長母音塊として保持する。[^30_1]
たとえば `su + u` は `Suu` のように表記する。[^30_1]

`CV + ai/oi/ui` は、原則として二重母音塊として保持する。[^30_1]
したがって `bai`、`kai` は `Bai`、`Kai` のように扱う。[^30_1]

### 5.4 独立母音化

`CV + V` または `CyV + V` の連接は、常に連母音として扱うのではなく、独立母音化する場合と連母音塊として維持する場合を分ける。[^30_1]
たとえば `siteori` は `SiTeORi`、`osaete` は `OSaETe`、`gyoukai` は `GyoUKai` のように表記されうる。[^30_1]

### 5.5 連母音維持

一部の母音連接は、独立母音へ分離せず塊として維持する。[^30_1]
たとえば `sieawo` は `SieaWo` のように、`siea` を一続きの塊として観察対象にできる。[^30_1]

## 6. 複合語と表記単位

中点・句読点・括弧・ハイフンを含む文字列は、必要に応じて「複合語」として1単位で扱う。[^30_1]
このとき、複合語は空白区切りの語より上位の表記単位として扱われる。[^30_1]

例外照合では、まず元の表記単位そのままで照合し、必要に応じて先頭・末尾の句読点を除いた正規化キーでも照合する。[^30_1]
これにより、前後の語が変わっても、複合語そのものに対して演出的例外を安定して適用できる。[^30_1]

## 7. 基本表示と表示例外

本規格では、基本表示と表示例外を明確に分離する。[^30_1]
基本表示は再束ね規則に基づく機械生成結果であり、表示例外は公開上の字面調整のための後段ルールである。[^30_1]

表示例外は、`display-exception-rules.json` 相当の辞書で管理する。[^30_1]
この辞書は、語単位にも複合語単位にも適用できる構成が望ましい。[^30_1]

## 8. 表示例

| 入力 | 基本表示または代表表示 | 備考 |
| :-- | :-- | :-- |
| `nippon` | `Nippon` [^30_1] | 語末 `n` は独立化しない。 |
| `kensaku` | `KenSaKu` [^30_1] | `ken` を `CVn` として束ねる。 |
| `enzin` | `EnZin` [^30_1] | `en` を `Vn` として束ねる。 |
| `toppuno` | `ToppuNo` [^30_1] | 促音を後続塊へ吸収する。 |
| `sieawo` | `SieaWo` [^30_1] | 連母音塊維持の例。 |
| `gyoukai` | `GyoUKai` [^30_1] | 独立母音化の例。 |
| `siteori` | `SiTeORi` [^30_1] | 独立母音化の例。 |
| `eigo・ratengo` | `EIGo・RaTenGo` [^30_1] | 複合語単位で例外を与える例。 |

## 9. 規格上の設計原則

すばる式ローマ字では、違和感は誤記ではなく、仮名塊を立たせるための演出である。[^30_1]
したがって、規格設計では「どこに視覚的ひっかかりを置くか」を、音読可能性と同じくらい重要な要件として扱う。[^30_1]

また、規格としては、すべてを語彙例外へ逃がすのではなく、できる限り再束ね規則として一般化し、そのうえでなお必要なものだけを表示例外に委ねるのが望ましい。[^30_1]
この分離があることで、表記法としての一貫性と、創作上の演出自由度とを両立しやすくなる。[^30_1]

## 10. 今後の拡張項目

今後の正式化では、少なくとも次を別表として明文化する必要がある。[^30_1]

- モーラ風トークン定義表。[^30_1]
- 再束ね優先順位表。[^30_1]
- 独立母音化／連母音維持の判定表。[^30_1]
- 複合語例外キーの正規化規則。[^30_1]
- 基本表示と表示例外の責務分担表。[^30_1]

本草案は、現行UIデモで合意・試行されている内容を仕様書の形へ並べ直した初期文書であり、正式版では「必須規則」「推奨規則」「演出例外」をさらに層別するのが望ましい。[^30_1]

# すばる式ローマ字 筆記法規格（草案）

本書は、訓令式ローマ字を基底入力として、仮名塊を視覚的に立ち上げる「すばる式ローマ字」の筆記法を定義するための草案である。[^32_1]
目的は、一般的な欧文単語と日本語ローマ字列とを初見の字面で見分けやすくし、仮名単位・仮名塊単位を強く意識させることにある。[^32_1]

## 1. 目的と適用範囲

すばる式ローマ字は、学校教育用の厳密な正書法そのものを置き換える規格ではなく、訓令式ローマ字の入力列をもとに、視覚的な違和感を利用して日本語の音節性を強調する筆記法として設計される。[^32_1]
そのため本規格は、音声転写の忠実さだけでなく、「どこをひと塊の仮名として見せるか」という表記設計を中核に置く。[^32_1]

適用対象は、半角小文字の訓令式ローマ字入力を基本とする日本語表記である。[^32_1]
公開表示では、基本変換による表示と、後段の表示例外とを分離して扱う構成を採る。[^32_1]

## 2. 基本原則

### 2.1 子音先頭の立ち上げ

子音を表す文字で始まる仮名単位は、先頭字のみを大文字化する。[^32_1]
したがって、通常の `CV` や `CyV` は `Ka`、`Si`、`Gyo` のように表記する。[^32_1]

### 2.2 独立母音の立ち上げ

子音を伴わない独立したア行母音は、大文字で表記する。[^32_1]
たとえば、再解釈上で独立母音と判定された `o` や `e` は `O`、`E` のように示す。[^32_1]

### 2.3 撥音 `ん`

撥音は小文字 `n` を基本とし、独立した大文字化対象とはしない。[^32_1]
直後が母音字または `y` の場合のみ `n'` とし、それ以外では `n` を用いる。[^32_1]

### 2.4 促音 `っ`

促音は入力解析上では独立トークンとして保持するが、公開表示では単独文字としては立てず、後続子音を肥大化させる要素として扱う。[^32_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入りの塊として再構成する。[^32_1]

## 3. 入力前提

入力は半角小文字の訓令式ローマ字とする。[^32_1]
空白は語境界として保持し、変換処理はまず語または表記単位の分割から始める。[^32_1]

句読点、中点、括弧、ハイフンを含む文字列は、必要に応じて単なる語ではなく「表記単位」として扱う。[^32_1]
これは、複合語や演出的なまとまりを、空白区切りより上位の単位として保持するためである。[^32_1]

### 3.1 分かち書きの推奨

すばる式ローマ字では、可読性と仮名塊の視認性を高めるため、語または表記単位ごとの分かち書きを推奨する。[^32_1]
とくに学習用・観察用・公開用の表示では、連続したローマ字列を詰めて書くよりも、意味上または表記上のまとまりごとに空白で区切る方が、仮名塊の境界と違和感の位置とを把握しやすい。[^32_1]

分かち書きは必須ではないが、本規格の運用上は推奨事項として扱う。[^32_1]
ただし、中点・句読点・括弧・ハイフンを含む複合語については、必要に応じて空白で分断せず、ひとまとまりの表記単位として保持してよい。[^32_1]

とくに、長音、促音、助詞は、分かち書きした方が視認性と教育効果の両面で有利であるため、優先的に区切ることを推奨する。[^32_1]
長音は仮名塊の持続感を、促音は子音肥大化の位置を、助詞は文法上の切れ目を、それぞれ空白によって見失いにくくできるためである。[^32_1]

たとえば、長音を含む語、促音を含む語、格助詞や係助詞を伴う連続列では、字面を詰めるよりも、`KaHanSuu No`、`To ppu No`、`Ni HonGo No` のように観察単位を明示する書き分けが推奨される。[^32_1]

## 4. 処理単位

本規格では、入力文字列を次の順で処理する。[^32_1]

1. 表記単位へ分割する。[^32_1]
2. 表記単位をモーラ風トークン列へ分解する。[^32_1]
3. トークン列を仮名塊へ再束ねする。[^32_1]
4. 基本表示を生成する。[^32_1]
5. 必要に応じて表示例外を適用する。[^32_1]

モーラ風トークンは、少なくとも `CV`、`CyV`、`N`、`Q`、`V` を基本分類とする。[^32_1]
ここで重要なのは、最終表示がトークン列の単純な逐次大文字化ではなく、再束ね後の仮名塊に基づいて決まる点である。[^32_1]

### 4.1 モーラ風トークン定義表

| トークン名 | 記号 | 定義 | 代表例 | 基本的な表示上の扱い |
| :-- | :-- | :-- | :-- | :-- |
| 母音単位 | `V` | 子音を伴わない単独母音。 | `a`, `i`, `u`, `e`, `o` [^32_1] | 独立母音と判定された場合は大文字で表示する。 [^32_1] |
| 子音+母音単位 | `CV` | 1子音+1母音から成る基本単位。 | `ka`, `si`, `te`, `no` [^32_1] | 先頭子音のみ大文字化し、`Ka`, `Si`, `Te`, `No` のように表示する。 [^32_1] |
| 拗音系単位 | `CyV` | 子音+`y`+母音から成る拗音系単位。 | `kyo`, `gyo`, `sya`, `tyu` [^32_1] | `CV` と同様に先頭のみ大文字化し、`Kyo`, `Gyo`, `Sya`, `Tyu` のように表示する。 [^32_1] |
| 撥音単位 | `N` | `ん` に対応する単位。直後が子音または語末のときに抽出される。 | `n` in `ken`, `han`, `en` [^32_1] | 単独大文字化はせず、再束ね時に `CVn` または `Vn` へ吸収する。 [^32_1] |
| 促音単位 | `Q` | 促音 `っ` に相当する重子音予告単位。 | `p` in `toppu`, `t` in `kitte` [^32_1] | 単独表示せず、後続 `CV` または `CyV` を肥大化させる。 [^32_1] |
| 未確定子音単位 | `C?` | 上記規則に収まらない孤立子音または暫定処理単位。 | 解析途中の単独子音 [^32_1] | 原則として暫定処理用とし、正式表示では再解釈または例外処理の対象とする。 [^32_1] |

この表は、入力解析段階での最小単位を定めるものであり、公開表示の最終単位そのものを定めるものではない。[^32_1]
最終的な字面は、これらのトークンが再束ね規則によって仮名塊へまとめ直された後に決定される。[^32_1]

## 5. 仮名塊再解釈

### 5.1 `CVn` / `Vn`

`CV + N + 子音` および `CV + N + 語末` は、`CVn` の一塊として再束ねする。[^32_1]
同様に、`V + N + 子音` および `V + N + 語末` は `Vn` の一塊として再束ねする。[^32_1]

この規則により、`ken` は `Ken`、`han` は `Han`、`en` は `En` のように表記される。[^32_1]
語末 `n` は独立クラスタを作らず、直前塊へ小文字従属させる。[^32_1]

### 5.2 `Q + CV` / `Q + CyV`

促音は単独表示せず、後続塊に吸収する。[^32_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入り塊へ、`Q + CyV` は `kkyo`、`ttya` のような塊へ再束ねする。[^32_1]

### 5.3 長母音と二重母音

`CV + 同母音V` は、長母音塊として保持する。[^32_1]
たとえば `su + u` は `Suu` のように表記する。[^32_1]

`CV + ai/oi/ui` は、原則として二重母音塊として保持する。[^32_1]
したがって `bai`、`kai` は `Bai`、`Kai` のように扱う。[^32_1]

### 5.4 独立母音化

`CV + V` または `CyV + V` の連接は、常に連母音として扱うのではなく、独立母音化する場合と連母音塊として維持する場合を分ける。[^32_1]
たとえば `siteori` は `SiTeORi`、`osaete` は `OSaETe`、`gyoukai` は `GyoUKai` のように表記されうる。[^32_1]

### 5.5 連母音維持

一部の母音連接は、独立母音へ分離せず塊として維持する。[^32_1]
たとえば `sieawo` は `SieaWo` のように、`siea` を一続きの塊として観察対象にできる。[^32_1]

## 6. 複合語と表記単位

中点・句読点・括弧・ハイフンを含む文字列は、必要に応じて「複合語」として1単位で扱う。[^32_1]
このとき、複合語は空白区切りの語より上位の表記単位として扱われる。[^32_1]

例外照合では、まず元の表記単位そのままで照合し、必要に応じて先頭・末尾の句読点を除いた正規化キーでも照合する。[^32_1]
これにより、前後の語が変わっても、複合語そのものに対して演出的例外を安定して適用できる。[^32_1]

## 7. 基本表示と表示例外

本規格では、基本表示と表示例外を明確に分離する。[^32_1]
基本表示は再束ね規則に基づく機械生成結果であり、表示例外は公開上の字面調整のための後段ルールである。[^32_1]

表示例外は、`display-exception-rules.json` 相当の辞書で管理する。[^32_1]
この辞書は、語単位にも複合語単位にも適用できる構成が望ましい。[^32_1]

## 8. 表示例

| 入力 | 基本表示または代表表示 | 備考 |
| :-- | :-- | :-- |
| `nippon` | `Nippon` [^32_1] | 語末 `n` は独立化しない。 |
| `kensaku` | `KenSaKu` [^32_1] | `ken` を `CVn` として束ねる。 |
| `enzin` | `EnZin` [^32_1] | `en` を `Vn` として束ねる。 |
| `toppuno` | `ToppuNo` [^32_1] | 促音を後続塊へ吸収する。 |
| `sieawo` | `SieaWo` [^32_1] | 連母音塊維持の例。 |
| `gyoukai` | `GyoUKai` [^32_1] | 独立母音化の例。 |
| `siteori` | `SiTeORi` [^32_1] | 独立母音化の例。 |
| `eigo・ratengo` | `EIGo・RaTenGo` [^32_1] | 複合語単位で例外を与える例。 |

## 9. 規格上の設計原則

すばる式ローマ字では、違和感は誤記ではなく、仮名塊を立たせるための演出である。[^32_1]
したがって、規格設計では「どこに視覚的ひっかかりを置くか」を、音読可能性と同じくらい重要な要件として扱う。[^32_1]

また、規格としては、すべてを語彙例外へ逃がすのではなく、できる限り再束ね規則として一般化し、そのうえでなお必要なものだけを表示例外に委ねるのが望ましい。[^32_1]
この分離があることで、表記法としての一貫性と、創作上の演出自由度とを両立しやすくなる。[^32_1]

## 10. 今後の拡張項目

今後の正式化では、少なくとも次を別表として明文化する必要がある。[^32_1]

- 再束ね優先順位表。[^32_1]
- 独立母音化／連母音維持の判定表。[^32_1]
- 複合語例外キーの正規化規則。[^32_1]
- 基本表示と表示例外の責務分担表。[^32_1]

本草案は、現行UIデモで合意・試行されている内容を仕様書の形へ並べ直した初期文書であり、正式版では「必須規則」「推奨規則」「演出例外」をさらに層別するのが望ましい。[^32_1]

# 再束ね優先順位表を作成して下さい。

再束ね優先順位は、いまの実装・説明から素直に整理すると、だいたい次の階層になります。

## 再束ね優先順位（概要）

1. 促音の吸収（`Q + CV / CyV`）
2. 撥音を含む終止塊（`CVn` / `Vn`）
3. 長母音・二重母音の維持（`CV + V`）
4. 独立母音化（`CV + V` / `CyV + V` → `V` 独立）
5. 残りの単純塊（単独 `CV` / `CyV` / `V`）

## 再束ね優先順位表

| 優先度 | 規則名 | 形式 | 例（入力） | 再束ね結果（塊） | 備考 |
| :-- | :-- | :-- | :-- | :-- | :-- |
| 1 | 促音吸収 | `Q + CV` / `Q + CyV` | `toppu`, `kitte`, `kkyo` | `Toppu`, `KiTTe`, `KKyo` のような重子音塊 | 促音は単独表示せず、最優先で後続塊に吸収する。子音の「つかえ」を字面でまとめて見せる段階。 |
| 2 | 撥音終止塊 | `CV + N` / `V + N` （後続が子音 or 語末） | `ken`, `han`, `en` | `Ken`, `Han`, `En` | 撥音を単独塊にせず、直前塊へ従属させて `CVn` / `Vn` を形成する。語末 `n` がバラけないよう、早い段階で束ねる。 |
| 3 | 長母音・二重母音維持 | `CV + 同母音V` / `CV + ai/oi/ui` | `su-u`, `kai`, `toi` | `Suu`, `Kai`, `Toi` | 長音や二重母音を「伸び」を持った一塊として優先的に保持する。ここではまだ独立母音化しない。 |
| 4 | 独立母音化 | `CV + V` / `CyV + V` のうち、独立母音と判断したもの | `siteori`, `osaete`, `gyoukai` | `SiTe-ORi`, `OSaE-Te`, `GyoU-Kai`（表記イメージ） | 連母音をすべて繋げるのではなく、意味的・視覚的に強調したい母音を切り出す。長音・二重母音維持より後段で、観察・演出目的に応じて適用する層。 |
| 5 | 単純塊保持 | 残った `CV` / `CyV` / `V` | `ka`, `si`, `kyo`, 単独 `a` など | `Ka`, `Si`, `Kyo`, `A` | 上記いずれにも当たらない単位は、そのまま一塊として保持し、大文字化規則に従って表示する。 |

※ ここでの再束ね結果の表記は説明用のイメージであり、実際の最終表示は仕様書の他条項（独立母音の扱いなど）との組み合わせで決まる前提です。

## この優先順位の意味

- 促音と撥音（1・2）は「分かれていると読みにくいもの」なので、早めに塊化してしまう層です。
- 長母音・二重母音（3）は、日本語としての音の自然さを保つための層。
- 独立母音化（4）は、すばる式ローマ字の「違和感演出」の中核で、**音の自然さより視覚的なひっかかりを優先的に設計する層**。
- 残り（5）は、通常のローマ字として破綻しない最低限の単位として扱われます。

もしよければ、この表を仕様書の「5. 仮名塊再解釈」の直後か、「別表 A: 再束ね優先順位」として差し込む形で整えましょうか。それとも、まずは「促音と長音のどちらを優先するか」など、もう少し細かい優先度の調整から詰めますか。

# 細かい優先度の調整から詰めます

はい、その段取りがよいです。[^34_1]

いまの仕様書を見る限り、再束ね候補として少なくとも `CVn / Vn`、`Q + CV / CyV`、長母音・二重母音、独立母音化、連母音維持が並んでおり、ここを**どの順で先に食うか**が規格の芯になります。[^34_1]
ここを曖昧にしたまま表だけ作ると、実装ごとに `Kai` と `KaI`、`Toppu` と `ToPpu` のような揺れが起きやすいです。[^34_1]

## 先に固定したい優先軸

細かい優先度を詰めるなら、まず次の4軸を固定するのが自然です。[^34_1]

- 促音吸収は、他の再束ねより先か。[^34_1]
- 撥音吸収 `CVn / Vn` は、長母音・独立母音化より先か。[^34_1]
- `CV + V` に遭遇したとき、長母音維持・二重母音維持・独立母音化のどれを優先するか。[^34_1]
- 連母音維持は一般規則か、限定規則か。[^34_1]


## いまの草案から見た妥当な順

現状の文言とこれまでのデモ挙動から見ると、暫定順は次が一番ぶれにくいです。[^34_1]

1. `Q + CV / Q + CyV` の促音吸収。[^34_1]
2. `CV + N` / `V + N` の撥音吸収。[^34_1]
3. `CV + 同母音V` の長母音維持。[^34_1]
4. `CV + ai/oi/ui` の二重母音維持。[^34_1]
5. 独立母音化の個別判定。[^34_1]
6. 連母音維持の個別判定。[^34_1]
7. 残余トークンの単純束ね。[^34_1]

この順にすると、「まず日本語として崩すと読みにくいもの」を先に固定し、その後で**すばる式特有の違和感演出**を足す構造になります。[^34_1]

## とくに争点になる箇所

一番揉めやすいのは、`gyoukai`、`kaihatu`、`seisyohou` のような **`CV/CyV + V` 連接** です。[^34_1]
ここは「長母音・二重母音として自然読みに寄せるか」「独立母音として違和感を立てるか」で字面が大きく変わるので、一般規則で吸う範囲と、例外へ逃がす範囲を切り分ける必要があります。[^34_1]

## 次の詰め方

次段では、再束ね優先順位表をいきなり清書するより、まず **衝突ケース一覧** を作るのがよいです。[^34_1]
具体的には `Q+CV`、`CV+N`、`CV+V`、`CyV+V` が競合しうる語を並べて、「この語はどの規則を先に適用するか」を表にすると、仕様書の優先順位表がそのまま固めやすくなります。[^34_1]

まずは、次の衝突ケース群を基準にしてよいと思います。[^34_1]

- `toppu`, `kitte`, `gakkou` — 促音優先の確認用。[^34_1]
- `ken`, `hon`, `sinya`, `kinyoubi` — 撥音優先と `n'` 条件の確認用。[^34_1]
- `kai`, `oi`, `suu`, `tou` — 長母音・二重母音優先の確認用。[^34_1]
- `siteori`, `osaete`, `gyoukai`, `seisyohou` — 独立母音化との競合確認用。[^34_1]

この4群で、まず「どの規則が勝つか」を決めるのが次の一手です。[^34_1]

# 促音優先の確認

促音優先は、**最優先でよい**と思います。[^35_1]

## 理由

仕様書草案でも、促音 `Q` は「単独表示せず、後続 `CV` または `CyV` を肥大化させる」とされており、これは他の再束ね候補より先に処理しないと字面が崩れやすいです。[^35_1]
たとえば `toppu` や `kitte` で促音を後回しにすると、`to + ppu` のような自然な塊ではなく、`ToPpu` のように不必要な切れ目が立ってしまい、仮名塊としてのまとまりが弱くなります。[^35_1]

## 規格文としての整理

したがって、再束ね優先順位では次のように置くのが自然です。[^35_1]

- **第1優先**: `Q + CV` / `Q + CyV` を先に吸収し、重子音入り塊を確定する。[^35_1]
- その後に `CVn / Vn`、長母音・二重母音、独立母音化へ進む。[^35_1]

この順にすると、促音は「後続子音の性質を決める前処理」として機能し、他の規則との競合をかなり減らせます。[^35_1]

## 実例での見え方

確認用の代表例は、次の3つで十分です。[^35_1]

- `toppu` → `Toppu`[^35_1]
- `kitte` → `KiTTe` ではなく、促音吸収後の一塊として扱う方針を取る。[^35_1]
- `gakkou` → まず `Q + ko` を束ね、その後で後続の長音・母音連接を判断する。[^35_1]

特に `gakkou` は、促音優先のあとに長母音・連母音判定が来るので、**促音と長音の競合確認用**として使いやすいです。[^35_1]

## 次の詰めどころ

次に決めるべきは、促音吸収後の塊を**完全一塊とみなすか**、それとも後段でさらに母音連接処理の対象に含めるか、です。[^35_1]
ここを決めると、`kitteiru` や `gakkou` のような語で、促音塊の後ろまで含めた再束ね範囲をどこまで許すかが明確になります。[^35_1]

# kitteiru や gakkou

↓
KitteIRu や GakkoU

その方向なら、促音吸収後の塊は**固定しつつ、後続の母音は別判定できる**、という整理が合っています。[^36_1]

## 解釈

`kitteiru` を `KitteIRu`、`gakkou` を `GakkoU` にしたいなら、`Q + CV` はまず `tte` や `kko` という**重子音入り塊として先に確定**し、その後ろの `i` や `u` は別の独立母音判定へ回す必要があります。[^36_1]
つまり、促音塊は「語全体を完全固定する塊」ではなく、**局所的に確定した仮名塊**として扱うのがよいです。[^36_1]

## 規則としての整理

この方針だと、優先順位は次のように読めます。[^36_1]

1. `Q + CV / Q + CyV` を最優先で再束ねする。[^36_1]
2. その結果できた塊の**後続トークン**については、長母音維持・独立母音化の判定を続行してよい。[^36_1]

この読みなら、`kitteiru` は `ki / tte / i / ru` のように観察でき、`KitteIRu` が自然に立ちます。[^36_1]
同様に `gakkou` も `ga / kko / u` と見て、`GakkoU` を許容できます。[^36_1]

## 仕様書に入れるなら

文言としては、5.2 の促音節に次の趣旨を足すとよさそうです。[^36_1]

- 促音吸収によって成立した塊は、その塊内部では固定とする。[^36_1]
- ただし、その塊の**直後に続く母音トークン**については、後段の長母音・独立母音化判定の対象としてよい。[^36_1]


## この方針の効果

この整理にすると、`ToppuNo` のような自然な促音塊も維持できる一方で、`KitteIRu` や `GakkoU` のような、すばる式らしい後段の母音立ち上げも両立できます。[^36_1]
要するに、促音優先は維持しつつ、**促音塊の後ろまではロックしない**、という設計です。[^36_1]

次はこの内容を、そのまま仕様書の 5.2 に追記して、あわせて 8. 表示例にも `kitteiru` と `gakkou` を追加する形で固めるとよいです。[^36_1]

# gyoukai、kaihatu、seisyohou

↓
GyoUKaI、KaIHaTu、SeISyoHoU

その3例を並べると、かなり大事な方針が見えます。[^37_1]

## 見えている傾向

`gyoukai → GyoUKaI`、`kaihatu → KaIHaTu`、`seisyohou → SeISyoHoU` という希望形は、単純な「長母音維持」でも「二重母音維持」でもなく、**母音連接ごとに独立母音化の位置を選ぶ**方針です。[^37_1]
つまり `ou` や `ai` を一律に塊保存するのではなく、語内のどこで視覚的なひっかかりを立てるかを優先している形です。[^37_1]

## ここで決めるべきこと

この3例を採るなら、少なくとも次の整理が必要です。[^37_1]

- `CyV + u` は、長音化より独立母音化を優先しうる。`gyo + u → GyoU`。[^37_1]
- `CV + ai` は、二重母音維持より分割を優先しうる。`ka + i → KaI`。[^37_1]
- 同じ語の中で、ある母音連接は分割し、別の母音連接は維持せず再び分割してよい。`sei + syo + hou → SeI + Syo + HoU` に近い考え方です。[^37_1]


## 暫定的な優先順位の読み替え

この方針に寄せるなら、前に置いた
「長母音・二重母音維持 → その後に独立母音化」
ではなく、次のようにしたほうが合います。[^37_1]

1. 促音吸収。[^37_1]
2. 撥音吸収。[^37_1]
3. **独立母音化判定。**[^37_1]
4. 独立母音化しなかったものだけ、長母音・二重母音として維持。[^37_1]
5. 残余単位を通常束ね。[^37_1]

この順なら、`gyoukai` は `gyo + u + ka + i` を `GyoUKaI` 系へ、`kaihatu` は `ka + i + ha + tu` を `KaIHaTu` へ、`seisyohou` は `se + i + syo + ho + u` を `SeISyoHoU` へ寄せやすくなります。[^37_1]

## ただし注意点

この方針は、自然読みよりも**仮名塊の視覚露出**を優先するので、一般ローマ字からのズレは強くなります。[^37_1]
そのぶん、すばる式ローマ字の目的にはかなっていますが、規格書では「独立母音化優先は本方式の意図的特徴である」と明記したほうがよいです。[^37_1]

## いま詰めるべき核心

要するに、争点は
**`CV/CyV + V` を見たとき、既定値は『維持』か『分割』か**
です。[^37_1]
今の3例を採るなら、既定値はかなり **分割寄り** になります。[^37_1]

ここで決め方は二つあります。[^37_1]

- A. 「すばる式では母音連接は原則分割」とする。[^37_1]
- B. 「特定条件だけ分割」とする。[^37_1]

この3例の字面を素直に取りに行くなら、現状は **A寄り** です。[^37_1]
次は、このA寄りを正式に採るか、それとも `ou / ai / ei` だけ条件分岐するBにするかを決める段です。[^37_1]

# ou / ai / ei だけ条件分岐するBにする根拠を教えて下さい

B案（`ou / ai / ei` だけ条件分岐する）を採る根拠は、だいたい次の三本柱で説明できます。[^38_1][^38_2][^38_3][^38_4]

## 1. 音声的に「特別扱いされている三種」だから

日本語の長音・二重母音の中でも、**`ou` と `ei` は実質「長母音」扱い**で説明されるのが一般的です。[^38_3][^38_4][^38_1]

- `ou` は ō（おー）、`ei` は ē（えー）として扱われることが多く、「o + u」「e + i」を分けて読むケースは少ない。[^38_4][^38_1][^38_3]
- `ai` は英語の eye に近い二重母音として説明され、「a から i への滑り」として一まとまりで教えられることが多い。[^38_2][^38_5][^38_6]

つまり、**音声的には「1拍を超えて伸びるまとまり」として認識されやすい3パターン**なので、他の連母音（例: `oe`, `ia` など）と同列に扱わず、「ここだけはまず自然なまとまりを尊重する」という分岐を立てるのは、かなり理屈が立ちます。[^38_1][^38_2][^38_3][^38_4]

## 2. 学習者がすでに「特別視」している組み合わせだから

ローマ字や日本語発音の入門解説では、長音・二重母音の表として `aa, ii, uu, ee, ei, oo, ou` のような一覧で、「`ei` は長いエ」「`ou` は長いオ」として、他の組み合わせから明示的に区別して教えるパターンが多いです。[^38_7][^38_3][^38_4][^38_1]
これにより学習者側の頭の中では、

- `ou` / `oo` → 伸びる「オー」
- `ei` / `ee` → 伸びる「エー」
- `ai`      → eye 型の二重母音

という「特別なまとまり」の枠が、すでにできています。[^38_2][^38_3][^38_4][^38_1]

すばる式ローマ字がこの三つだけを優先的に「維持側」扱いにするのは、**既存の学習経験と正面衝突しない最低限の配慮**として、かなり合理的です。

## 3. 表記上のノイズを増やし過ぎない「安全な閾値」だから

すばる式ローマ字の目的は、

- 「なんでもかんでも分割してギザギザにする」ことではなく
- 「仮名塊が欧文語から浮き上がる程度の違和感」を狙うこと

でしたよね。

そこで

- すべての `CV + V` を原則分割（A案）にすると、`toukyou` まで `ToUkyO` 方向に崩れてしまい、**ノイズ量が過剰**になります。
- 一方、`ou / ai / ei` を「まず自然な一塊として扱う優先候補」にしておけば、
    - `toukyou` のような典型語は自然寄りの `TouKyoU` 側にも寄せやすい
    - それ以外の `eo`, `ia`, `oe` などで思い切って独立母音化・分割を遊べる

というバランスが取れます。

つまり **B案は、「ここまでは日本語の耳にとって自然」に寄せておいて、その外側で思い切ってすばる式の違和感を暴れさせるための安全弁**として機能します。

***

まとめると、

- 音声学的にも教育的にも、`ou / ai / ei` は「特別扱いされている三種」である。[^38_3][^38_4][^38_1][^38_2]
- そこだけ維持側に回すと、既存のローマ字感覚と大きくぶつからずに済む。
- その代わり、その他の連母音ではすばる式ローマ字らしい「分割・独立母音化」の遊びが思い切ってできる。

この三点が、「`ou / ai / ei` だけ条件分岐するB案」を採る根拠になります。

この前提で、`gyoukai / kaihatu / seisyohou` に対しては、「`yoU` / `aI` / `eI` をそれぞれどう扱うか」をもう少し細かいルール（例: 語末の `ou` は長音優先、中間の `ai` は演出優先、など）として切り出す、という進め方が筋がよさそうです。

# 細かいルール（例: 語末の ou は長音優先、中間の ai は演出優先、など）として切り出します

はい、その切り出し方が一番よいです。[^39_1]

いまの仕様書だと、5.3 の「長母音と二重母音」と 5.4 の「独立母音化」がまだ少し広すぎるので、ここを**母音連接の条件分岐規則**として細文化すると、規格としてかなり強くなります。[^39_1]

## まず固定するとよい基本方針

現段階の合意を素直に整理すると、母音連接は一律処理ではなく、少なくとも `ou / ai / ei` を優先的な判定対象にするB案で進めるのが自然です。[^39_1]
その上で、「語末か語中か」「後続が子音か母音か」「演出上の切れ目を立てたいか」で分岐させると、`GakkoU`、`KaIHaTu`、`SeISyoHoU` のような希望形を説明しやすくなります。[^39_1]

## 細かいルールのたたき台

いったん仕様化しやすい形にすると、次のように置けます。[^39_1]

- `ou`: 語末または塊末では、長音維持よりも**独立母音化を優先**して `...oU` を許す。[^39_1]
- `ai`: 語中では、二重母音維持よりも**演出上の切れ目を優先**して `...aI...` を許す。[^39_1]
- `ei`: 語中・語末ともに、長音扱いに固定せず、**独立母音化を優先候補**として `...eI...` を許す。[^39_1]

この読みなら、`gyoukai → GyoUKaI`、`kaihatu → KaIHaTu`、`seisyohou → SeISyoHoU` は、かなり一貫した方針で説明できます。[^39_1]

## 3語への当てはめ

- `gyoukai` は、`gyo + u` で `ou` 相当の後半を独立母音化し、さらに `ka + i` で `ai` を演出分割して `GyoUKaI` と読む整理です。[^39_1]
- `kaihatu` は、語頭 `ka + i` を二重母音維持より分割優先として `KaIHaTu` にする整理です。[^39_1]
- `seisyohou` は、`se + i` を `SeI`、`ho + u` を `HoU` として、語中 `ei` と語末 `ou` の双方で独立母音化を採る整理です。[^39_1]


## 仕様書に入れるなら

文書上は、5.3 と 5.4 の間に、たとえば **「5.3.1 母音連接の条件分岐」** を立てるのがよいです。[^39_1]
そこに「`ou / ai / ei` は優先判定対象であり、長音・二重母音として固定せず、位置条件と演出条件に応じて独立母音化を選択できる」と書けば、規格らしい骨が通ります。[^39_1]

## 次に詰めるべき一点

細則化するなら、次は
**`ou / ai / ei` の既定値を『維持』にするか『分割許容』にするか**
を、規格文で明文化する段です。[^39_1]
いまの希望例を見る限り、かなり **分割許容を既定寄りに置く** のが合っています。[^39_1]

次はこの3種について、

- 語頭
- 語中
- 語末
の位置別に、既定表示を表にすると詰めやすいです。[^39_1]

# 位置別に、既定表示を表を作成して下さい

位置別の既定表示を、`ou / ai / ei` それぞれについて整理します。
ここでは「原則こうする」「ただし演出でこう崩せる」という二段階で書いてあります。

## `ou` の位置別・既定表示

| 位置 | 既定の扱い | 例 | 備考 |
| :-- | :-- | :-- | :-- |
| 語頭 | 滅多に出ないので、個別判断（基本は分割） | `ouji` → `O UZi` など | 語頭の長音は視覚的に強いので、原則「独立母音化寄り」で考える。 |
| 語中 | 原則「長音寄り」、ただし演出上分割可 | `toukyou` → `TouKyoU` / 演出で `ToU KyoU` | 読みやすさ優先なら ō 扱い、違和感を強めたい箇所だけ分割を許容。 |
| 語末 | **独立母音化優先** | `gakkou` → `GakkoU` | 語末の伸びを強調したいので、`oU` のように末尾母音を立てるのを既定値にする。 |

## `ai` の位置別・既定表示

| 位置 | 既定の扱い | 例 | 備考 |
| :-- | :-- | :-- | :-- |
| 語頭 | 二重母音としてひと塊 or 分割、作品ごとに方針決め | `aiko` → `AiKo` / `A IKo` | 語頭 `Ai` は固有名詞的に見えやすいので、世界観に合わせて決める。 |
| 語中 | **演出優先で分割寄り** | `kaihatu` → `KaIHaTu` | もっとも「違和感を立てやすい」ポジションなので、既定を分割寄りにしておくと設計しやすい。 |
| 語末 | 文脈依存（感嘆・間投詞は維持寄り） | `yasai` → `YaSai` / 演出で `YaSaI` | 語末 `ai` は感情語で多く、滑らかさを残したい場合も多いので、ここだけは維持寄りを残しておく。 |

## `ei` の位置別・既定表示

| 位置 | 既定の扱い | 例 | 備考 |
| :-- | :-- | :-- | :-- |
| 語頭 | 原則分割寄り（長いエを強調） | `eiji` → `EIzi` | 長いエで始まる字面を強く出したいので、`EI` のように頭から立てる方向を既定にしやすい。 |
| 語中 | **独立母音化優先** | `seisyohou` → `SeISyoHoU` | 「長いエ」として一塊にするより、`eI` で切って違和感を出したいコア領域。 |
| 語末 | 独立母音化寄りだが、作品単位で統一 | `orei` → `OReI` | 語末 `ei` も「伸び」を感じさせたい位置なので、`eI` を既定寄りにしておくとすばる式らしい。 |

## 全体方針の要約

- `ou` は「語末だけは積極的に `oU` を許容する」、語中は読みやすさと演出の間で揺らせる。
- `ai` は「語中は演出優先で分割寄り」、語末は語感次第で二重母音維持も残す。
- `ei` は「語中・語末とも独立母音化寄り」がすばる式らしい芯になる。

仕様書に落とすときは、上の3表を「母音連接の位置別既定値」として載せたうえで、「この既定値からの逸脱は、作品・用語ごとの表示例外として管理する」と一文入れておくと、ルールと演出の境界がきれいに分かれます。

この既定値、まずは「すばる式としてのデフォルト」として固定してしまってよさそうですか、それとも特に迷っている組み合わせ（語末 `ai` など）がありますか。

# yasai → YaSaI　に　これ以外は「すばる式としてのデフォルト」として固定でOK

了解です。[^41_1]

## 確定する既定値

- `yasai` は **`YaSaI`**。[^41_1]
- それ以外の `ou / ai / ei` の位置別既定表示は、先ほどの整理を**すばる式としてのデフォルト**として固定する。[^41_1]


## この決め方の意味

これで `ai` だけが全面的に分割寄りになるのではなく、**語末 `ai` には維持寄りを残さず、`YaSaI` を既定値として採る**ことになります。[^41_1]
つまり、母音連接全体としては「分割許容を強く持つ規格」だが、その中でも `ai` は語末まで含めて演出側へ倒す、というかなり明快な方針になります。[^41_1]

## 仕様書へ落とすなら

仕様書上は、母音連接の位置別既定表示を表にして、`ai` の語末欄を
**「独立母音化寄り。既定表示は `...aI`」**
と書けば十分です。[^41_1]
その例示として `yasai → YaSaI` を載せると、規格のニュアンスがかなり伝わりやすくなります。[^41_1]

## 次の自然な作業

次はこの確定方針を、そのまま仕様書本文へ正式追記する段です。[^41_1]
あわせて表示例に `yasai → YaSaI`、`gakkou → GakkoU`、`kitteiru → KitteIRu`、`kaihatu → KaIHaTu`、`seisyohou → SeISyoHoU` を追加すると、かなり規格書らしく締まります。[^41_1]

# すばる式ローマ字 筆記法規格（草案）

本書は、訓令式ローマ字を基底入力として、仮名塊を視覚的に立ち上げる「すばる式ローマ字」の筆記法を定義するための草案である。[^42_1]
目的は、一般的な欧文単語と日本語ローマ字列とを初見の字面で見分けやすくし、仮名単位・仮名塊単位を強く意識させることにある。[^42_1]

## 1. 目的と適用範囲

すばる式ローマ字は、学校教育用の厳密な正書法そのものを置き換える規格ではなく、訓令式ローマ字の入力列をもとに、視覚的な違和感を利用して日本語の音節性を強調する筆記法として設計される。[^42_1]
そのため本規格は、音声転写の忠実さだけでなく、「どこをひと塊の仮名として見せるか」という表記設計を中核に置く。[^42_1]

適用対象は、半角小文字の訓令式ローマ字入力を基本とする日本語表記である。[^42_1]
公開表示では、基本変換による表示と、後段の表示例外とを分離して扱う構成を採る。[^42_1]

## 2. 基本原則

### 2.1 子音先頭の立ち上げ

子音を表す文字で始まる仮名単位は、先頭字のみを大文字化する。[^42_1]
したがって、通常の `CV` や `CyV` は `Ka`、`Si`、`Gyo` のように表記する。[^42_1]

### 2.2 独立母音の立ち上げ

子音を伴わない独立したア行母音は、大文字で表記する。[^42_1]
たとえば、再解釈上で独立母音と判定された `o` や `e` は `O`、`E` のように示す。[^42_1]

### 2.3 撥音 `ん`

撥音は小文字 `n` を基本とし、独立した大文字化対象とはしない。[^42_1]
直後が母音字または `y` の場合のみ `n'` とし、それ以外では `n` を用いる。[^42_1]

### 2.4 促音 `っ`

促音は入力解析上では独立トークンとして保持するが、公開表示では単独文字としては立てず、後続子音を肥大化させる要素として扱う。[^42_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入りの塊として再構成する。[^42_1]

## 3. 入力前提

入力は半角小文字の訓令式ローマ字とする。[^42_1]
空白は語境界として保持し、変換処理はまず語または表記単位の分割から始める。[^42_1]

句読点、中点、括弧、ハイフンを含む文字列は、必要に応じて単なる語ではなく「表記単位」として扱う。[^42_1]
これは、複合語や演出的なまとまりを、空白区切りより上位の単位として保持するためである。[^42_1]

### 3.1 分かち書きの推奨

すばる式ローマ字では、可読性と仮名塊の視認性を高めるため、語または表記単位ごとの分かち書きを推奨する。[^42_1]
とくに学習用・観察用・公開用の表示では、連続したローマ字列を詰めて書くよりも、意味上または表記上のまとまりごとに空白で区切る方が、仮名塊の境界と違和感の位置とを把握しやすい。[^42_1]

分かち書きは必須ではないが、本規格の運用上は推奨事項として扱う。[^42_1]
ただし、中点・句読点・括弧・ハイフンを含む複合語については、必要に応じて空白で分断せず、ひとまとまりの表記単位として保持してよい。[^42_1]

とくに、長音、促音、助詞は、分かち書きした方が視認性と教育効果の両面で有利であるため、優先的に区切ることを推奨する。[^42_1]
長音は仮名塊の持続感を、促音は子音肥大化の位置を、助詞は文法上の切れ目を、それぞれ空白によって見失いにくくできるためである。[^42_1]

たとえば、長音を含む語、促音を含む語、格助詞や係助詞を伴う連続列では、字面を詰めるよりも、`KaHanSuu No`、`To ppu No`、`Ni HonGo No` のように観察単位を明示する書き分けが推奨される。[^42_1]

## 4. 処理単位

本規格では、入力文字列を次の順で処理する。[^42_1]

1. 表記単位へ分割する。[^42_1]
2. 表記単位をモーラ風トークン列へ分解する。[^42_1]
3. トークン列を仮名塊へ再束ねする。[^42_1]
4. 基本表示を生成する。[^42_1]
5. 必要に応じて表示例外を適用する。[^42_1]

モーラ風トークンは、少なくとも `CV`、`CyV`、`N`、`Q`、`V` を基本分類とする。[^42_1]
ここで重要なのは、最終表示がトークン列の単純な逐次大文字化ではなく、再束ね後の仮名塊に基づいて決まる点である。[^42_1]

### 4.1 モーラ風トークン定義表

| トークン名 | 記号 | 定義 | 代表例 | 基本的な表示上の扱い |
| :-- | :-- | :-- | :-- | :-- |
| 母音単位 | `V` | 子音を伴わない単独母音。 | `a`, `i`, `u`, `e`, `o` [^42_1] | 独立母音と判定された場合は大文字で表示する。 [^42_1] |
| 子音+母音単位 | `CV` | 1子音+1母音から成る基本単位。 | `ka`, `si`, `te`, `no` [^42_1] | 先頭子音のみ大文字化し、`Ka`, `Si`, `Te`, `No` のように表示する。 [^42_1] |
| 拗音系単位 | `CyV` | 子音+`y`+母音から成る拗音系単位。 | `kyo`, `gyo`, `sya`, `tyu` [^42_1] | `CV` と同様に先頭のみ大文字化し、`Kyo`, `Gyo`, `Sya`, `Tyu` のように表示する。 [^42_1] |
| 撥音単位 | `N` | `ん` に対応する単位。直後が子音または語末のときに抽出される。 | `n` in `ken`, `han`, `en` [^42_1] | 単独大文字化はせず、再束ね時に `CVn` または `Vn` へ吸収する。 [^42_1] |
| 促音単位 | `Q` | 促音 `っ` に相当する重子音予告単位。 | `p` in `toppu`, `t` in `kitte` [^42_1] | 単独表示せず、後続 `CV` または `CyV` を肥大化させる。 [^42_1] |
| 未確定子音単位 | `C?` | 上記規則に収まらない孤立子音または暫定処理単位。 | 解析途中の単独子音 [^42_1] | 原則として暫定処理用とし、正式表示では再解釈または例外処理の対象とする。 [^42_1] |

この表は、入力解析段階での最小単位を定めるものであり、公開表示の最終単位そのものを定めるものではない。[^42_1]
最終的な字面は、これらのトークンが再束ね規則によって仮名塊へまとめ直された後に決定される。[^42_1]

## 5. 仮名塊再解釈

### 5.1 `CVn` / `Vn`

`CV + N + 子音` および `CV + N + 語末` は、`CVn` の一塊として再束ねする。[^42_1]
同様に、`V + N + 子音` および `V + N + 語末` は `Vn` の一塊として再束ねする。[^42_1]

この規則により、`ken` は `Ken`、`han` は `Han`、`en` は `En` のように表記される。[^42_1]
語末 `n` は独立クラスタを作らず、直前塊へ小文字従属させる。[^42_1]

### 5.2 `Q + CV` / `Q + CyV`

促音は単独表示せず、後続塊に吸収する。[^42_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入り塊へ、`Q + CyV` は `kkyo`、`ttya` のような塊へ再束ねする。[^42_1]

### 5.3 長母音と二重母音

`CV + 同母音V` は、長母音塊として保持する。[^42_1]
たとえば `su + u` は `Suu` のように表記する。[^42_1]

`CV + ai/oi/ui` は、二重母音候補として扱うが、すばる式ローマ字では常に維持するとは限らない。[^42_1]
とくに `ou`、`ai`、`ei` については、位置条件と演出条件に応じた既定表示を別に定める。[^42_1]

### 5.3.1 母音連接の位置別既定表示

すばる式ローマ字では、`ou`、`ai`、`ei` を優先判定対象とし、一律に長母音塊または二重母音塊へ固定しない。[^42_1]
これらは語頭・語中・語末の位置によって、長母音維持より独立母音化を優先する場合がある。[^42_1]


| 連接 | 位置 | 既定表示 | 代表例 | 取扱い方針 |
| :-- | :-- | :-- | :-- | :-- |
| `ou` | 語頭 | 分割寄り | `ouji` → `OUZi` [^42_1] | 語頭では独立母音の立ち上がりを優先する。 [^42_1] |
| `ou` | 語中 | 維持寄りだが分割許容 | `toukyou` → `TouKyoU` [^42_1] | 可読性を優先するが、演出上は独立母音化を許容する。 [^42_1] |
| `ou` | 語末 | 独立母音化優先 | `gakkou` → `GakkoU` [^42_1] | 語末長音は `...oU` の形で立てるのを既定とする。 [^42_1] |
| `ai` | 語頭 | 分割寄り | `aiko` → `AiKo` または `AIKo` の検討対象 [^42_1] | 語頭 `ai` は作品ごとの調整余地を残す。 [^42_1] |
| `ai` | 語中 | 独立母音化優先 | `kaihatu` → `KaIHaTu` [^42_1] | 語中 `ai` は演出上の切れ目を立てる既定とする。 [^42_1] |
| `ai` | 語末 | 独立母音化優先 | `yasai` → `YaSaI` [^42_1] | 語末 `ai` も `...aI` を既定表示とする。 [^42_1] |
| `ei` | 語頭 | 独立母音化優先 | `eizi` → `EIZi` [^42_1] | 語頭 `ei` は `EI` として立てる方向を既定とする。 [^42_1] |
| `ei` | 語中 | 独立母音化優先 | `seisyohou` → `SeISyoHoU` [^42_1] | 語中 `ei` は長音化より `eI` を優先する。 [^42_1] |
| `ei` | 語末 | 独立母音化優先 | `orei` → `OReI` [^42_1] | 語末 `ei` も `...eI` を既定表示とする。 [^42_1] |

上表に定めた既定表示は、すばる式ローマ字における標準的な既定値として扱う。[^42_1]
ただし、作品固有名詞、複合語、視覚演出上の要請がある場合は、表示例外によって調整してよい。[^42_1]

### 5.4 独立母音化

`CV + V` または `CyV + V` の連接は、常に連母音として扱うのではなく、独立母音化する場合と連母音塊として維持する場合を分ける。[^42_1]
たとえば `siteori` は `SiTeORi`、`osaete` は `OSaETe`、`gyoukai` は `GyoUKaI` のように表記されうる。[^42_1]
また、`kitteiru` は `KitteIRu`、`gakkou` は `GakkoU` のように、促音塊確定後の後続母音に対しても独立母音化を適用できる。[^42_1]

### 5.5 連母音維持

一部の母音連接は、独立母音へ分離せず塊として維持する。[^42_1]
たとえば `sieawo` は `SieaWo` のように、`siea` を一続きの塊として観察対象にできる。[^42_1]

## 6. 複合語と表記単位

中点・句読点・括弧・ハイフンを含む文字列は、必要に応じて「複合語」として1単位で扱う。[^42_1]
このとき、複合語は空白区切りの語より上位の表記単位として扱われる。[^42_1]

例外照合では、まず元の表記単位そのままで照合し、必要に応じて先頭・末尾の句読点を除いた正規化キーでも照合する。[^42_1]
これにより、前後の語が変わっても、複合語そのものに対して演出的例外を安定して適用できる。[^42_1]

## 7. 基本表示と表示例外

本規格では、基本表示と表示例外を明確に分離する。[^42_1]
基本表示は再束ね規則に基づく機械生成結果であり、表示例外は公開上の字面調整のための後段ルールである。[^42_1]

表示例外は、`display-exception-rules.json` 相当の辞書で管理する。[^42_1]
この辞書は、語単位にも複合語単位にも適用できる構成が望ましい。[^42_1]

## 9. 規格上の設計原則

すばる式ローマ字では、違和感は誤記ではなく、仮名塊を立たせるための演出である。[^42_1]
したがって、規格設計では「どこに視覚的ひっかかりを置くか」を、音読可能性と同じくらい重要な要件として扱う。[^42_1]

また、規格としては、すべてを語彙例外へ逃がすのではなく、できる限り再束ね規則として一般化し、そのうえでなお必要なものだけを表示例外に委ねるのが望ましい。[^42_1]
この分離があることで、表記法としての一貫性と、創作上の演出自由度とを両立しやすくなる。[^42_1]

## 10. 今後の拡張項目

今後の正式化では、少なくとも次を別表として明文化する必要がある。[^42_1]

- 再束ね優先順位表。[^42_1]
- 独立母音化／連母音維持の判定表。[^42_1]
- 複合語例外キーの正規化規則。[^42_1]
- 基本表示と表示例外の責務分担表。[^42_1]

本草案は、現行UIデモで合意・試行されている内容を仕様書の形へ並べ直した初期文書であり、正式版では「必須規則」「推奨規則」「演出例外」をさらに層別するのが望ましい。[^42_1]

# 衝突ケース

    ken, hon, sinya, kinyoubi — 撥音優先と n' 条件の確認
    この衝突ケース群では、論点はかなりはっきりしています。[^43_1]

## 結論

- `ken`、`hon` は **撥音優先** でよいです。[^43_1]
- `sinya`、`kinyoubi` は **撥音優先ではなく、`n'` 条件を優先確認する別ケース** として分けて扱うのがよいです。[^43_1]

つまり、この4語は全部を同じ箱に入れるより、
**`CVn / Vn` 確定組** と **`n'` 分岐確認組**
に分けたほうが仕様がきれいになります。[^43_1]

## 1. `ken`, `hon`

仕様書の 5.1 では、`CV + N + 子音` および `CV + N + 語末` は `CVn / Vn` として再束ねすると定めています。[^43_1]
そのため `ken` は `Ken`、`hon` は `Hon` とし、`n` を独立させないのが既定で問題ありません。[^43_1]

- `ken` → `Ken`[^43_1]
- `hon` → `Hon`[^43_1]

この2語は、**撥音が直前塊へ吸収される標準例**です。[^43_1]

## 2. `sinya`

`sinya` は、`n` の直後が `y` なので、仕様書の基本原則 2.3 にある **`n'` 条件** に入ります。[^43_1]
したがって、ここでは `CVn` を優先するのではなく、**`si + n' + ya`** と読ませる側を優先すべきです。[^43_1]

つまり既定は、

- `sinya` → `Sin'Ya` 方向[^43_1]

と整理するのが自然です。[^43_1]

ここで `SinYa` のようにアポストロフィ無しでまとめると、`sinya` が `si-nya` なのか `sin-ya` なのかが視覚上ぶれやすくなります。[^43_1]

## 3. `kinyoubi`

`kinyoubi` も同じで、`n` の直後が `y` なので `n'` 条件の確認例です。[^43_1]
しかもこの語は、その後ろに `you` が続くため、**撥音処理と母音連接処理が連続する複合衝突例**としてかなり重要です。[^43_1]

既定の読み筋は、

- `ki + n' + yo + u + bi`[^43_1]
- 表示は `Kin'YoUBi` 方向[^43_1]

とするのが、今の規格方針と一番整合します。[^43_1]

ここでは、まず `n'` を確定してから、後続の `yo + u` に対して語中 `ou` 相当の位置別既定表示を当てる、という順番になります。[^43_1]

## 衝突整理としての表

| 語 | 主要衝突 | 先に勝つ規則 | 既定表示 |
| :-- | :-- | :-- | :-- |
| `ken` | `CV + N + 語末` | 撥音吸収 `CVn` [^43_1] | `Ken` [^43_1] |
| `hon` | `CV + N + 語末` | 撥音吸収 `CVn` [^43_1] | `Hon` [^43_1] |
| `sinya` | `N` vs `n' + y` | `n'` 条件 [^43_1] | `Sin'Ya` [^43_1] |
| `kinyoubi` | `N` vs `n' + y`、その後 `you` | `n'` 条件 → 後続母音判定 [^43_1] | `Kin'YoUBi` [^43_1] |

## 規格文としての含意

ここから言えるのは、**撥音吸収 `CVn / Vn` は強い規則だが、直後が母音または `y` の場合には適用しない**、ということです。[^43_1]
つまり優先順位としては、

1. `n` の直後が母音または `y` か判定する。
2. そうなら `n'` を立てる。
3. そうでなければ `CVn / Vn` へ吸収する。
という順で書くのが一番明快です。[^43_1]

次はこの内容を、そのまま仕様書の 2.3 と 5.1 に補強文として入れ、表示例に `sinya → Sin'Ya` と `kinyoubi → Kin'YoUBi` を追加すると、撥音の節がかなり締まります。[^43_1]

# すばる式ローマ字 筆記法規格（草案）

本書は、訓令式ローマ字を基底入力として、仮名塊を視覚的に立ち上げる「すばる式ローマ字」の筆記法を定義するための草案である。[^44_1]
目的は、一般的な欧文単語と日本語ローマ字列とを初見の字面で見分けやすくし、仮名単位・仮名塊単位を強く意識させることにある。[^44_1]

## 1. 目的と適用範囲

すばる式ローマ字は、学校教育用の厳密な正書法そのものを置き換える規格ではなく、訓令式ローマ字の入力列をもとに、視覚的な違和感を利用して日本語の音節性を強調する筆記法として設計される。[^44_1]
そのため本規格は、音声転写の忠実さだけでなく、「どこをひと塊の仮名として見せるか」という表記設計を中核に置く。[^44_1]

適用対象は、半角小文字の訓令式ローマ字入力を基本とする日本語表記である。[^44_1]
公開表示では、基本変換による表示と、後段の表示例外とを分離して扱う構成を採る。[^44_1]

## 2. 基本原則

### 2.1 子音先頭の立ち上げ

子音を表す文字で始まる仮名単位は、先頭字のみを大文字化する。[^44_1]
したがって、通常の `CV` や `CyV` は `Ka`、`Si`、`Gyo` のように表記する。[^44_1]

### 2.2 独立母音の立ち上げ

子音を伴わない独立したア行母音は、大文字で表記する。[^44_1]
たとえば、再解釈上で独立母音と判定された `o` や `e` は `O`、`E` のように示す。[^44_1]

### 2.3 撥音 `ん`

撥音は小文字 `n` を基本とし、独立した大文字化対象とはしない。[^44_1]
直後が母音字または `y` の場合のみ `n'` とし、それ以外では `n` を用いる。[^44_1]
したがって、撥音を `CVn` または `Vn` として直前塊へ吸収する規則は、直後が母音字または `y` の場合には適用しない。[^44_1]
この場合は、まず `n'` を確定し、その後で後続の母音連接や拗音系単位の再束ね判定へ進む。[^44_1]

### 2.4 促音 `っ`

促音は入力解析上では独立トークンとして保持するが、公開表示では単独文字としては立てず、後続子音を肥大化させる要素として扱う。[^44_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入りの塊として再構成する。[^44_1]

## 3. 入力前提

入力は半角小文字の訓令式ローマ字とする。[^44_1]
空白は語境界として保持し、変換処理はまず語または表記単位の分割から始める。[^44_1]

句読点、中点、括弧、ハイフンを含む文字列は、必要に応じて単なる語ではなく「表記単位」として扱う。[^44_1]
これは、複合語や演出的なまとまりを、空白区切りより上位の単位として保持するためである。[^44_1]

### 3.1 分かち書きの推奨

すばる式ローマ字では、可読性と仮名塊の視認性を高めるため、語または表記単位ごとの分かち書きを推奨する。[^44_1]
とくに学習用・観察用・公開用の表示では、連続したローマ字列を詰めて書くよりも、意味上または表記上のまとまりごとに空白で区切る方が、仮名塊の境界と違和感の位置とを把握しやすい。[^44_1]

分かち書きは必須ではないが、本規格の運用上は推奨事項として扱う。[^44_1]
ただし、中点・句読点・括弧・ハイフンを含む複合語については、必要に応じて空白で分断せず、ひとまとまりの表記単位として保持してよい。[^44_1]

とくに、長音、促音、助詞は、分かち書きした方が視認性と教育効果の両面で有利であるため、優先的に区切ることを推奨する。[^44_1]
長音は仮名塊の持続感を、促音は子音肥大化の位置を、助詞は文法上の切れ目を、それぞれ空白によって見失いにくくできるためである。[^44_1]

たとえば、長音を含む語、促音を含む語、格助詞や係助詞を伴う連続列では、字面を詰めるよりも、`KaHanSuu No`、`To ppu No`、`Ni HonGo No` のように観察単位を明示する書き分けが推奨される。[^44_1]

## 4. 処理単位

本規格では、入力文字列を次の順で処理する。[^44_1]

1. 表記単位へ分割する。[^44_1]
2. 表記単位をモーラ風トークン列へ分解する。[^44_1]
3. トークン列を仮名塊へ再束ねする。[^44_1]
4. 基本表示を生成する。[^44_1]
5. 必要に応じて表示例外を適用する。[^44_1]

モーラ風トークンは、少なくとも `CV`、`CyV`、`N`、`Q`、`V` を基本分類とする。[^44_1]
ここで重要なのは、最終表示がトークン列の単純な逐次大文字化ではなく、再束ね後の仮名塊に基づいて決まる点である。[^44_1]

### 4.1 モーラ風トークン定義表

| トークン名 | 記号 | 定義 | 代表例 | 基本的な表示上の扱い |
| :-- | :-- | :-- | :-- | :-- |
| 母音単位 | `V` | 子音を伴わない単独母音。 | `a`, `i`, `u`, `e`, `o` [^44_1] | 独立母音と判定された場合は大文字で表示する。 [^44_1] |
| 子音+母音単位 | `CV` | 1子音+1母音から成る基本単位。 | `ka`, `si`, `te`, `no` [^44_1] | 先頭子音のみ大文字化し、`Ka`, `Si`, `Te`, `No` のように表示する。 [^44_1] |
| 拗音系単位 | `CyV` | 子音+`y`+母音から成る拗音系単位。 | `kyo`, `gyo`, `sya`, `tyu` [^44_1] | `CV` と同様に先頭のみ大文字化し、`Kyo`, `Gyo`, `Sya`, `Tyu` のように表示する。 [^44_1] |
| 撥音単位 | `N` | `ん` に対応する単位。直後が子音または語末のときに抽出される。 | `n` in `ken`, `han`, `en` [^44_1] | 単独大文字化はせず、再束ね時に `CVn` または `Vn` へ吸収する。 [^44_1] |
| 促音単位 | `Q` | 促音 `っ` に相当する重子音予告単位。 | `p` in `toppu`, `t` in `kitte` [^44_1] | 単独表示せず、後続 `CV` または `CyV` を肥大化させる。 [^44_1] |
| 未確定子音単位 | `C?` | 上記規則に収まらない孤立子音または暫定処理単位。 | 解析途中の単独子音 [^44_1] | 原則として暫定処理用とし、正式表示では再解釈または例外処理の対象とする。 [^44_1] |

この表は、入力解析段階での最小単位を定めるものであり、公開表示の最終単位そのものを定めるものではない。[^44_1]
最終的な字面は、これらのトークンが再束ね規則によって仮名塊へまとめ直された後に決定される。[^44_1]

## 5. 仮名塊再解釈

### 5.1 `CVn` / `Vn`

`CV + N + 子音` および `CV + N + 語末` は、`CVn` の一塊として再束ねする。[^44_1]
同様に、`V + N + 子音` および `V + N + 語末` は `Vn` の一塊として再束ねする。[^44_1]

この規則により、`ken` は `Ken`、`han` は `Han`、`en` は `En` のように表記される。[^44_1]
語末 `n` は独立クラスタを作らず、直前塊へ小文字従属させる。[^44_1]
ただし、直後が母音字または `y` の場合は、本規則より `n'` 条件を優先する。[^44_1]
そのため `sinya` は `Sin'Ya`、`kinyoubi` は `Kin'YoUBi` のように扱い、`n` を `CVn` として直前塊へ吸収しない。[^44_1]

### 5.2 `Q + CV` / `Q + CyV`

促音は単独表示せず、後続塊に吸収する。[^44_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入り塊へ、`Q + CyV` は `kkyo`、`ttya` のような塊へ再束ねする。[^44_1]

### 5.3 長母音と二重母音

`CV + 同母音V` は、長母音塊として保持する。[^44_1]
たとえば `su + u` は `Suu` のように表記する。[^44_1]

`CV + ai/oi/ui` は、二重母音候補として扱うが、すばる式ローマ字では常に維持するとは限らない。[^44_1]
とくに `ou`、`ai`、`ei` については、位置条件と演出条件に応じた既定表示を別に定める。[^44_1]

### 5.3.1 母音連接の位置別既定表示

すばる式ローマ字では、`ou`、`ai`、`ei` を優先判定対象とし、一律に長母音塊または二重母音塊へ固定しない。[^44_1]
これらは語頭・語中・語末の位置によって、長母音維持より独立母音化を優先する場合がある。[^44_1]


| 連接 | 位置 | 既定表示 | 代表例 | 取扱い方針 |
| :-- | :-- | :-- | :-- | :-- |
| `ou` | 語頭 | 分割寄り | `ouji` → `OUZi` [^44_1] | 語頭では独立母音の立ち上がりを優先する。 [^44_1] |
| `ou` | 語中 | 維持寄りだが分割許容 | `toukyou` → `TouKyoU` [^44_1] | 可読性を優先するが、演出上は独立母音化を許容する。 [^44_1] |
| `ou` | 語末 | 独立母音化優先 | `gakkou` → `GakkoU` [^44_1] | 語末長音は `...oU` の形で立てるのを既定とする。 [^44_1] |
| `ai` | 語頭 | 分割寄り | `aiko` → `AiKo` または `AIKo` の検討対象 [^44_1] | 語頭 `ai` は作品ごとの調整余地を残す。 [^44_1] |
| `ai` | 語中 | 独立母音化優先 | `kaihatu` → `KaIHaTu` [^44_1] | 語中 `ai` は演出上の切れ目を立てる既定とする。 [^44_1] |
| `ai` | 語末 | 独立母音化優先 | `yasai` → `YaSaI` [^44_1] | 語末 `ai` も `...aI` を既定表示とする。 [^44_1] |
| `ei` | 語頭 | 独立母音化優先 | `eizi` → `EIZi` [^44_1] | 語頭 `ei` は `EI` として立てる方向を既定とする。 [^44_1] |
| `ei` | 語中 | 独立母音化優先 | `seisyohou` → `SeISyoHoU` [^44_1] | 語中 `ei` は長音化より `eI` を優先する。 [^44_1] |
| `ei` | 語末 | 独立母音化優先 | `orei` → `OReI` [^44_1] | 語末 `ei` も `...eI` を既定表示とする。 [^44_1] |

上表に定めた既定表示は、すばる式ローマ字における標準的な既定値として扱う。[^44_1]
ただし、作品固有名詞、複合語、視覚演出上の要請がある場合は、表示例外によって調整してよい。[^44_1]

### 5.4 独立母音化

`CV + V` または `CyV + V` の連接は、常に連母音として扱うのではなく、独立母音化する場合と連母音塊として維持する場合を分ける。[^44_1]
たとえば `siteori` は `SiTeORi`、`osaete` は `OSaETe`、`gyoukai` は `GyoUKaI` のように表記されうる。[^44_1]
また、`kitteiru` は `KitteIRu`、`gakkou` は `GakkoU` のように、促音塊確定後の後続母音に対しても独立母音化を適用できる。[^44_1]

### 5.5 連母音維持

一部の母音連接は、独立母音へ分離せず塊として維持する。[^44_1]
たとえば `sieawo` は `SieaWo` のように、`siea` を一続きの塊として観察対象にできる。[^44_1]

## 6. 複合語と表記単位

中点・句読点・括弧・ハイフンを含む文字列は、必要に応じて「複合語」として1単位で扱う。[^44_1]
このとき、複合語は空白区切りの語より上位の表記単位として扱われる。[^44_1]

例外照合では、まず元の表記単位そのままで照合し、必要に応じて先頭・末尾の句読点を除いた正規化キーでも照合する。[^44_1]
これにより、前後の語が変わっても、複合語そのものに対して演出的例外を安定して適用できる。[^44_1]

## 7. 基本表示と表示例外

本規格では、基本表示と表示例外を明確に分離する。[^44_1]
基本表示は再束ね規則に基づく機械生成結果であり、表示例外は公開上の字面調整のための後段ルールである。[^44_1]

表示例外は、`display-exception-rules.json` 相当の辞書で管理する。[^44_1]
この辞書は、語単位にも複合語単位にも適用できる構成が望ましい。[^44_1]

## 9. 規格上の設計原則

すばる式ローマ字では、違和感は誤記ではなく、仮名塊を立たせるための演出である。[^44_1]
したがって、規格設計では「どこに視覚的ひっかかりを置くか」を、音読可能性と同じくらい重要な要件として扱う。[^44_1]

また、規格としては、すべてを語彙例外へ逃がすのではなく、できる限り再束ね規則として一般化し、そのうえでなお必要なものだけを表示例外に委ねるのが望ましい。[^44_1]
この分離があることで、表記法としての一貫性と、創作上の演出自由度とを両立しやすくなる。[^44_1]

## 10. 今後の拡張項目

今後の正式化では、少なくとも次を別表として明文化する必要がある。[^44_1]

- 再束ね優先順位表。[^44_1]
- 独立母音化／連母音維持の判定表。[^44_1]
- 複合語例外キーの正規化規則。[^44_1]
- 基本表示と表示例外の責務分担表。[^44_1]

本草案は、現行UIデモで合意・試行されている内容を仕様書の形へ並べ直した初期文書であり、正式版では「必須規則」「推奨規則」「演出例外」をさらに層別するのが望ましい。[^44_1]

# 衝突ケース

    kai, oi, suu, tou — 長母音・二重母音優先の確認
    この4例は、長母音・二重母音の扱いをかなりきれいに分けて確認できます。[^45_1]

## 結論

- `suu` は **長母音維持優先** でよいです。[^45_1]
- `kai` は **既定として独立母音化優先** でよいです。[^45_1]
- `tou` は **語末 `ou` の独立母音化優先** でよいです。[^45_1]
- `oi` は、現時点では **一般の二重母音候補として維持寄り** に置くのが安全です。[^45_1]

つまりこの衝突ケース群は、
**同母音長音は維持、`ou / ai` は条件分岐、`oi` はまだ一般側に残す**
という整理になります。[^45_1]

## 1. `suu`

仕様書 5.3 では、`CV + 同母音V` は長母音塊として保持するとしています。[^45_1]
したがって `suu` は、そのまま **`Suu`** を既定で問題ありません。[^45_1]

これは、もっとも素直な長母音維持の確認例です。[^45_1]

## 2. `kai`

仕様書 5.3.1 では、`ai` は語中・語末とも独立母音化優先に寄せており、`yasai → YaSaI` や `kaihatu → KaIHaTu` がその既定例です。[^45_1]
この流れに合わせるなら、単独語 `kai` も **`KaI`** を既定表示にするのが自然です。[^45_1]

ここで `Kai` に戻すと、語末 `ai` を独立母音化する既定方針とぶつかります。[^45_1]

## 3. `tou`

仕様書では、`ou` は語末で独立母音化優先とし、`gakkou → GakkoU` を既定例にしています。[^45_1]
したがって単独語 `tou` も、同じ方針で **`ToU`** を既定にするのが整合的です。[^45_1]

つまり `tou` は、「長いオー」を自然に流すより、**語末の `U` を立たせる** 例として扱うことになります。[^45_1]

## 4. `oi`

`oi` については、現行仕様書の 5.3 に `CV + ai/oi/ui` は二重母音候補とありつつ、5.3.1 で特別に条件分岐を立てているのは `ou / ai / ei` だけです。[^45_1]
したがって現段階では、`oi` はその特別分岐に入れず、**一般の二重母音候補として維持寄り** に置くのが一番きれいです。[^45_1]

既定表示は **`Oi`** でよく、ここを `OI` にまで倒すと、今度は `oi` も特別扱いする別ルールが要ります。[^45_1]

## 衝突整理表

| 語 | 主要衝突 | 先に勝つ規則 | 既定表示 |
| :-- | :-- | :-- | :-- |
| `suu` | 同母音長音 vs 独立母音化 | 長母音維持 [^45_1] | `Suu` [^45_1] |
| `kai` | `ai` 二重母音 vs 独立母音化 | `ai` の位置別既定 [^45_1] | `KaI` [^45_1] |
| `tou` | `ou` 長音 vs 独立母音化 | 語末 `ou` の既定 [^45_1] | `ToU` [^45_1] |
| `oi` | `oi` 二重母音 vs 独立母音化 | 一般二重母音維持 [^45_1] | `Oi` [^45_1] |

## 規格文としての含意

この4例から言えるのは、**すばる式ローマ字はすべての二重母音を同じように崩すわけではない**、という点です。[^45_1]
特別分岐は `ou / ai / ei` に限定し、`oi` や `ui` は当面一般則側へ残すことで、規格全体のノイズ量を制御できます。[^45_1]

次はこの内容を、そのまま仕様書の 5.3 または 5.3.1 に補強文として入れ、表示例に `kai → KaI`、`oi → Oi`、`suu → Suu`、`tou → ToU` を足すと、長母音・二重母音節がかなり締まります。[^45_1]

# すばる式ローマ字 筆記法規格（草案）

本書は、訓令式ローマ字を基底入力として、仮名塊を視覚的に立ち上げる「すばる式ローマ字」の筆記法を定義するための草案である。[^46_1]
目的は、一般的な欧文単語と日本語ローマ字列とを初見の字面で見分けやすくし、仮名単位・仮名塊単位を強く意識させることにある。[^46_1]

## 1. 目的と適用範囲

すばる式ローマ字は、学校教育用の厳密な正書法そのものを置き換える規格ではなく、訓令式ローマ字の入力列をもとに、視覚的な違和感を利用して日本語の音節性を強調する筆記法として設計される。[^46_1]
そのため本規格は、音声転写の忠実さだけでなく、「どこをひと塊の仮名として見せるか」という表記設計を中核に置く。[^46_1]

適用対象は、半角小文字の訓令式ローマ字入力を基本とする日本語表記である。[^46_1]
公開表示では、基本変換による表示と、後段の表示例外とを分離して扱う構成を採る。[^46_1]

## 2. 基本原則

### 2.1 子音先頭の立ち上げ

子音を表す文字で始まる仮名単位は、先頭字のみを大文字化する。[^46_1]
したがって、通常の `CV` や `CyV` は `Ka`、`Si`、`Gyo` のように表記する。[^46_1]

### 2.2 独立母音の立ち上げ

子音を伴わない独立したア行母音は、大文字で表記する。[^46_1]
たとえば、再解釈上で独立母音と判定された `o` や `e` は `O`、`E` のように示す。[^46_1]

### 2.3 撥音 `ん`

撥音は小文字 `n` を基本とし、独立した大文字化対象とはしない。[^46_1]
直後が母音字または `y` の場合のみ `n'` とし、それ以外では `n` を用いる。[^46_1]
したがって、撥音を `CVn` または `Vn` として直前塊へ吸収する規則は、直後が母音字または `y` の場合には適用しない。[^46_1]
この場合は、まず `n'` を確定し、その後で後続の母音連接や拗音系単位の再束ね判定へ進む。[^46_1]

### 2.4 促音 `っ`

促音は入力解析上では独立トークンとして保持するが、公開表示では単独文字としては立てず、後続子音を肥大化させる要素として扱う。[^46_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入りの塊として再構成する。[^46_1]

## 3. 入力前提

入力は半角小文字の訓令式ローマ字とする。[^46_1]
空白は語境界として保持し、変換処理はまず語または表記単位の分割から始める。[^46_1]

句読点、中点、括弧、ハイフンを含む文字列は、必要に応じて単なる語ではなく「表記単位」として扱う。[^46_1]
これは、複合語や演出的なまとまりを、空白区切りより上位の単位として保持するためである。[^46_1]

### 3.1 分かち書きの推奨

すばる式ローマ字では、可読性と仮名塊の視認性を高めるため、語または表記単位ごとの分かち書きを推奨する。[^46_1]
とくに学習用・観察用・公開用の表示では、連続したローマ字列を詰めて書くよりも、意味上または表記上のまとまりごとに空白で区切る方が、仮名塊の境界と違和感の位置とを把握しやすい。[^46_1]

分かち書きは必須ではないが、本規格の運用上は推奨事項として扱う。[^46_1]
ただし、中点・句読点・括弧・ハイフンを含む複合語については、必要に応じて空白で分断せず、ひとまとまりの表記単位として保持してよい。[^46_1]

とくに、長音、促音、助詞は、分かち書きした方が視認性と教育効果の両面で有利であるため、優先的に区切ることを推奨する。[^46_1]
長音は仮名塊の持続感を、促音は子音肥大化の位置を、助詞は文法上の切れ目を、それぞれ空白によって見失いにくくできるためである。[^46_1]

たとえば、長音を含む語、促音を含む語、格助詞や係助詞を伴う連続列では、字面を詰めるよりも、`KaHanSuu No`、`To ppu No`、`Ni HonGo No` のように観察単位を明示する書き分けが推奨される。[^46_1]

## 4. 処理単位

本規格では、入力文字列を次の順で処理する。[^46_1]

1. 表記単位へ分割する。[^46_1]
2. 表記単位をモーラ風トークン列へ分解する。[^46_1]
3. トークン列を仮名塊へ再束ねする。[^46_1]
4. 基本表示を生成する。[^46_1]
5. 必要に応じて表示例外を適用する。[^46_1]

モーラ風トークンは、少なくとも `CV`、`CyV`、`N`、`Q`、`V` を基本分類とする。[^46_1]
ここで重要なのは、最終表示がトークン列の単純な逐次大文字化ではなく、再束ね後の仮名塊に基づいて決まる点である。[^46_1]

### 4.1 モーラ風トークン定義表

| トークン名 | 記号 | 定義 | 代表例 | 基本的な表示上の扱い |
| :-- | :-- | :-- | :-- | :-- |
| 母音単位 | `V` | 子音を伴わない単独母音。 | `a`, `i`, `u`, `e`, `o` [^46_1] | 独立母音と判定された場合は大文字で表示する。 [^46_1] |
| 子音+母音単位 | `CV` | 1子音+1母音から成る基本単位。 | `ka`, `si`, `te`, `no` [^46_1] | 先頭子音のみ大文字化し、`Ka`, `Si`, `Te`, `No` のように表示する。 [^46_1] |
| 拗音系単位 | `CyV` | 子音+`y`+母音から成る拗音系単位。 | `kyo`, `gyo`, `sya`, `tyu` [^46_1] | `CV` と同様に先頭のみ大文字化し、`Kyo`, `Gyo`, `Sya`, `Tyu` のように表示する。 [^46_1] |
| 撥音単位 | `N` | `ん` に対応する単位。直後が子音または語末のときに抽出される。 | `n` in `ken`, `han`, `en` [^46_1] | 単独大文字化はせず、再束ね時に `CVn` または `Vn` へ吸収する。 [^46_1] |
| 促音単位 | `Q` | 促音 `っ` に相当する重子音予告単位。 | `p` in `toppu`, `t` in `kitte` [^46_1] | 単独表示せず、後続 `CV` または `CyV` を肥大化させる。 [^46_1] |
| 未確定子音単位 | `C?` | 上記規則に収まらない孤立子音または暫定処理単位。 | 解析途中の単独子音 [^46_1] | 原則として暫定処理用とし、正式表示では再解釈または例外処理の対象とする。 [^46_1] |

この表は、入力解析段階での最小単位を定めるものであり、公開表示の最終単位そのものを定めるものではない。[^46_1]
最終的な字面は、これらのトークンが再束ね規則によって仮名塊へまとめ直された後に決定される。[^46_1]

## 5. 仮名塊再解釈

### 5.1 `CVn` / `Vn`

`CV + N + 子音` および `CV + N + 語末` は、`CVn` の一塊として再束ねする。[^46_1]
同様に、`V + N + 子音` および `V + N + 語末` は `Vn` の一塊として再束ねする。[^46_1]

この規則により、`ken` は `Ken`、`han` は `Han`、`en` は `En` のように表記される。[^46_1]
語末 `n` は独立クラスタを作らず、直前塊へ小文字従属させる。[^46_1]
ただし、直後が母音字または `y` の場合は、本規則より `n'` 条件を優先する。[^46_1]
そのため `sinya` は `Sin'Ya`、`kinyoubi` は `Kin'YoUBi` のように扱い、`n` を `CVn` として直前塊へ吸収しない。[^46_1]

### 5.2 `Q + CV` / `Q + CyV`

促音は単独表示せず、後続塊に吸収する。[^46_1]
そのため `Q + CV` は `ppu`、`tta` のような重子音入り塊へ、`Q + CyV` は `kkyo`、`ttya` のような塊へ再束ねする。[^46_1]

### 5.3 長母音と二重母音

`CV + 同母音V` は、長母音塊として保持する。[^46_1]
たとえば `su + u` は `Suu` のように表記する。[^46_1]

`CV + ai/oi/ui` は、二重母音候補として扱うが、すばる式ローマ字では常に維持するとは限らない。[^46_1]
ただし、特別な位置別既定表示を設けるのは `ou`、`ai`、`ei` に限り、`oi` および `ui` は当面一般の二重母音候補として維持寄りに扱う。[^46_1]
したがって、`suu` は `Suu`、`oi` は `Oi` のように、同母音長音および一般二重母音は基本的に自然読みに寄せて保持する。[^46_1]
とくに `ou`、`ai`、`ei` については、位置条件と演出条件に応じた既定表示を別に定める。[^46_1]

### 5.3.1 母音連接の位置別既定表示

すばる式ローマ字では、`ou`、`ai`、`ei` を優先判定対象とし、一律に長母音塊または二重母音塊へ固定しない。[^46_1]
これらは語頭・語中・語末の位置によって、長母音維持より独立母音化を優先する場合がある。[^46_1]


| 連接 | 位置 | 既定表示 | 代表例 | 取扱い方針 |
| :-- | :-- | :-- | :-- | :-- |
| `ou` | 語頭 | 分割寄り | `ouji` → `OUZi` [^46_1] | 語頭では独立母音の立ち上がりを優先する。 [^46_1] |
| `ou` | 語中 | 維持寄りだが分割許容 | `toukyou` → `TouKyoU` [^46_1] | 可読性を優先するが、演出上は独立母音化を許容する。 [^46_1] |
| `ou` | 語末 | 独立母音化優先 | `gakkou` → `GakkoU` [^46_1] | 語末長音は `...oU` の形で立てるのを既定とする。 [^46_1] |
| `ai` | 語頭 | 分割寄り | `aiko` → `AiKo` または `AIKo` の検討対象 [^46_1] | 語頭 `ai` は作品ごとの調整余地を残す。 [^46_1] |
| `ai` | 語中 | 独立母音化優先 | `kaihatu` → `KaIHaTu` [^46_1] | 語中 `ai` は演出上の切れ目を立てる既定とする。 [^46_1] |
| `ai` | 語末 | 独立母音化優先 | `yasai` → `YaSaI` [^46_1] | 語末 `ai` も `...aI` を既定表示とする。 [^46_1] |
| `ei` | 語頭 | 独立母音化優先 | `eizi` → `EIZi` [^46_1] | 語頭 `ei` は `EI` として立てる方向を既定とする。 [^46_1] |
| `ei` | 語中 | 独立母音化優先 | `seisyohou` → `SeISyoHoU` [^46_1] | 語中 `ei` は長音化より `eI` を優先する。 [^46_1] |
| `ei` | 語末 | 独立母音化優先 | `orei` → `OReI` [^46_1] | 語末 `ei` も `...eI` を既定表示とする。 [^46_1] |

上表に定めた既定表示は、すばる式ローマ字における標準的な既定値として扱う。[^46_1]
そのため、`kai` は `KaI`、`tou` は `ToU` を既定とする一方、`oi` は `Oi`、`suu` は `Suu` を既定とする。[^46_1]
ただし、作品固有名詞、複合語、視覚演出上の要請がある場合は、表示例外によって調整してよい。[^46_1]

### 5.4 独立母音化

`CV + V` または `CyV + V` の連接は、常に連母音として扱うのではなく、独立母音化する場合と連母音塊として維持する場合を分ける。[^46_1]
たとえば `siteori` は `SiTeORi`、`osaete` は `OSaETe`、`gyoukai` は `GyoUKaI` のように表記されうる。[^46_1]
また、`kitteiru` は `KitteIRu`、`gakkou` は `GakkoU` のように、促音塊確定後の後続母音に対しても独立母音化を適用できる。[^46_1]

### 5.5 連母音維持

一部の母音連接は、独立母音へ分離せず塊として維持する。[^46_1]
たとえば `sieawo` は `SieaWo` のように、`siea` を一続きの塊として観察対象にできる。[^46_1]

## 6. 複合語と表記単位

中点・句読点・括弧・ハイフンを含む文字列は、必要に応じて「複合語」として1単位で扱う。[^46_1]
このとき、複合語は空白区切りの語より上位の表記単位として扱われる。[^46_1]

例外照合では、まず元の表記単位そのままで照合し、必要に応じて先頭・末尾の句読点を除いた正規化キーでも照合する。[^46_1]
これにより、前後の語が変わっても、複合語そのものに対して演出的例外を安定して適用できる。[^46_1]

## 7. 基本表示と表示例外

本規格では、基本表示と表示例外を明確に分離する。[^46_1]
基本表示は再束ね規則に基づく機械生成結果であり、表示例外は公開上の字面調整のための後段ルールである。[^46_1]

表示例外は、`display-exception-rules.json` 相当の辞書で管理する。[^46_1]
この辞書は、語単位にも複合語単位にも適用できる構成が望ましい。[^46_1]

## 9. 規格上の設計原則

すばる式ローマ字では、違和感は誤記ではなく、仮名塊を立たせるための演出である。[^46_1]
したがって、規格設計では「どこに視覚的ひっかかりを置くか」を、音読可能性と同じくらい重要な要件として扱う。[^46_1]

また、規格としては、すべてを語彙例外へ逃がすのではなく、できる限り再束ね規則として一般化し、そのうえでなお必要なものだけを表示例外に委ねるのが望ましい。[^46_1]
この分離があることで、表記法としての一貫性と、創作上の演出自由度とを両立しやすくなる。[^46_1]

## 10. 今後の拡張項目

今後の正式化では、少なくとも次を別表として明文化する必要がある。[^46_1]

- 再束ね優先順位表。[^46_1]
- 独立母音化／連母音維持の判定表。[^46_1]
- 複合語例外キーの正規化規則。[^46_1]
- 基本表示と表示例外の責務分担表。[^46_1]

本草案は、現行UIデモで合意・試行されている内容を仕様書の形へ並べ直した初期文書であり、正式版では「必須規則」「推奨規則」「演出例外」をさらに層別するのが望ましい。[^46_1]
