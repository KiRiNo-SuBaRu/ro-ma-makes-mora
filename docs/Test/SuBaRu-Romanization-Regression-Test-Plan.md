# 回帰テスト計画

## 第 1 章 総則

### 1.1  本テスト計画の目的

　本テスト計画は、「すばる式ローマ字 モーラ強調表現ガイドライン
Ver 2.2-draft1」に基づく標準モード表示仕様について、
バージョンアップ時に挙動が変化してはいけない箇所と
意図的に変化させた箇所を明示し、体系的に回帰テストすることを目的とする。

　対象は、ガイドライン本文のうち 5〜8 章で定義された
モーラ構築・再束ね・基本表示・代表語・表示例、および
9.2 / 9.4 節で示された拡張仕様ポリシーとする。

---

### 1.2 対象となるガイドラインのバージョン（例: Ver 2.2-draft1）

　本テスト計画は、すばる式ローマ字 モーラ強調表現ガイドライン
Ver 2.2-draft1 を基準バージョンとして策定する。

　Ver 2.1 系の挙動は、必要に応じて「legacy 参考値」として
三段ログや 7.3 節の履歴情報にのみ保持し、
current 環境の必須回帰テスト期待値は Ver 2.2 系に従う

---

### 1.3 テストブロック構成（詳細版ドラフト）

1.3 テストブロック構成

本テスト計画では、標準モード表示仕様に対する回帰テストを、
次の 5 種のテストブロックに分割して管理する。

    Block CV: CV / CyV 基本表示ブロック

    Block NQ: N / Q（撥音・促音）ブロック

    Block V: V（長母音・二重母音・V 特例）ブロック

    Block SP: 分かち書き・複合語基本表示ブロック

    Block EX: 表示例外・複合語例外キーブロック

各ブロックは、テスト対象となるルール群・代表語セット・
関連ガイドライン章・バージョン分岐の有無を明示する。

#### 1.3.1 Block CV: CV / CyV 基本表示

    対象ルール

        2.1 節「子音先頭の立ち上げ」に基づく CV / CyV の大文字化。

        4 章のトークン定義表における CV / CyV。

    代表語セット（例）

        ka, si, te, no, kyo, gyo, sya, tyu 等。

    主な検証内容

        先頭子音のみ大文字化され、それ以外の文字は小文字で維持されること。

        他ブロックの処理（長音・撥音・促音・分かち書き等）が
        介在しても、CV / CyV の基本表示ルールが崩れないこと。

---

#### 1.3.2 Block NQ: N / Q（撥音・促音）

    対象ルール

        5.1 節 CVn / Vn（撥音再束ね）

        5.2 節 Q + CV / Q + CyV（促音吸収）

    代表語セット（例）

        N: ken, en, enzin, sinya, kinyoubi

        Q: toppu, kitte, toppuno, kitteiru, gakkou

    主な検証内容

        CVn / Vn 形成と n' 条件の優先順位。

        促音が単独表示されず、常に後続の CV / CyV に吸収されること。

        促音吸収後も、長音・独立母音化など V 系ルールが
        後段で正しく適用されること（特に gakkou, kitteiru）。

---

#### 1.3.3 Block V: V（長母音・二重母音・V 特例）

    対象ルール

        5.3 節「長母音と二重母音」

        5.3.1 節「母音連接の位置別既定表示」

        5.4 節 代表語セット（V-eo / V-ae / V-ea を含む）

        7.2 節 V 代表語一覧

        7.3 節 V 例外拡張一覧（特に sieawo）

    代表語セット（例）

        NQ 章と重複しない範囲で: suu, oi, tou,
        yasai, gyoukai, kaihatu, seisyohou,
        siteori, osaete, sieawo 等。

    ブロック内サブ構成

        Block V-A: V-basic-and-long（現行標準値）

        Block V-B: V-sieawo-versioned（SieaWo → SieAWo のバージョン差）

        Block V-C: V-placeholder-guards（将来拡張用 placeholder の非使用チェック）

    主な検証内容

        長母音・一般二重母音の基本挙動。

        ou / ai / ei の位置別既定表示。

        V-eo / V-ae / V-ea 特例の適用範囲と、標準仕様／履歴仕様の切り分け。

---

#### 1.3.4 Block SP: 分かち書き・複合語基本表示

    対象ルール

        3.1 節「分かち書きの推奨」

        6 章「複合語と表記単位」

    代表語・代表列（例）

        分かち書き: KaHanSuu No, Toppu No, Ni HonGo No

        複合語: eigo・ratengo, kai-hatu 等。

    主な検証内容

        分かち書きした場合でも、語単位の基本表示ルールが維持されること。

        中点・ハイフンを含む複合語が 1 表記単位として扱われ、
        内部の N / Q / V 処理が安定していること。

---

#### 1.3.5 Block EX: 表示例外・複合語例外キー

    対象ルール

        別表 C. 複合語例外キーの正規化規則

        別表 D. 基本表示と表示例外の責務分担表

        7 章 表示例外の説明

        8 章 表示例（eigo・ratengo 等）

    代表例（例）

        表示例外有り: eigo・ratengo → EIGo・RaTenGo

        正規化キー確認: 「eigo・ratengo」, (seisyohou)

        非例外ガード: gyoukai, siteori 等。

    主な検証内容

        基本表示と表示例外の境界が仕様どおりに保たれていること。

        正規化キー生成が例外辞書と正しく連携していること。

        例外を持たない語に誤って表示例外が適用されないこと。。

---

### 1.4 1.4 仕様書との対応関係

    各テストブロックは、ガイドラインの特定章と 1 対 1 または 1 対 N の関係を持つ。

        Block CV: 主に 2.1 節・4 章（トークン定義）

        Block NQ: 5.1 / 5.2 節、および 8 章 NQ 表示例

        Block V: 5.3 / 5.3.1 / 5.4 節、7.2 / 7.3 節、8 章 V 表示例

        Block SP: 3.1 節・6 章・8 章の分かち書き・複合語例

        Block EX: 別表 C / 別表 D・7 章・8 章の表示例外

    ガイドライン本文に差分が生じた場合は、
    対応するブロックのテストケースおよび期待値も必ず更新するものとする。

---

### 1.5 仕様担当と実装担当の役割分担

    仕様担当の責務

        本テスト計画のテストケース ID・期待値・関連章リンクを維持・更新する。

        拡張仕様（extension / candidate / history）の採否や昇格・廃止を決めた際には、
        必ず Block V / EX など該当ブロックのテストケースを更新する。

    実装担当の責務

        本テスト計画のテストケース ID を、そのまま自動テスト等の
        ケース ID として採用し、pass/fail を報告する。

        新たな仕様提案や拡張提案を行う場合には、
        どのブロックにテストを追加すべきかの案を提案票に含める

---

### 1.6 バージョン分岐（legacy / current）の扱い

    原則として、すべてのブロックは「current 環境（Ver 2.2 系）」を
    前提とした期待値を定義する。

    一部の仕様（例: sieawo の V-ea 挙動など）については、
    legacy 挙動を追跡するための versioned テストケースを別途定義し、
    current 環境用の必須回帰セットとはブロックを分けて管理する。
